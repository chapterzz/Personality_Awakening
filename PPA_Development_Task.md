## 1. 项目概述与目标

本项目旨在构建一个云端实时互联的人格探索平台，服务于初中生群体。核心目标是打破本地数据孤岛，实现学生端（移动端优先）与教师端（大屏可视化）的**低延迟、近实时**数据同步；**端到端 SLA** 以 KPI 与 NFR 为准（学生提交 → 教师可见 &lt; 1s）。系统需具备游戏化测试体验（AVG 模式）、严格的隐私保护机制（匿名化）以及科学的心理学科普内容。

### 核心成功指标 (KPIs)
- [ ] **实时性**：端到端实时延时（学生提交 -> 教师可见）&lt; 1s。
- [ ] **参与度**：学生主动分享率 > 60%。
- [ ] **完成率**：测试流程完成率 > 90%。
- [ ] **兼容性**：完美适配 iOS/Android 主流浏览器及桌面端。

### 2.1 研发节奏、测试规划与门禁

- **小步快跑（仅指节奏）**：在每周 Phase 内，按任务 ID 再拆**可交付小切片**（例如「T1.2 完成迁移」「T1.4 进度 API 可测」）。**每一片开发完成后，先执行测试与门禁，全部通过后再开始下一片**；发现问题**先修复再推进**。
- **测试规划（须预先完备）**：**类型、矩阵、工具、门禁命令**以 **`PPA_Tech_SPEC.md` §5** 为准；**每个环节须有单元测试**（api：`Jest`；web：`Vitest`），不得以「切片小」为由推迟到 Week 6 再补。
- **门禁命令（根目录）**：日常合并前至少 **`pnpm test:gate`**（`gate:static` + `test:unit` + **`test:integration`**（api supertest））；涉及页面/跨服务时加 **`pnpm test:e2e`**；详见技术规格书 §5.2 / §5.4。
- **与 Week 6 的关系**：**T5.x**（压力、安全、全链路 E2E、部署）为**收口与报告**；**T5.2** 要求 Playwright 全链路 **100% 通过**，与 §5.3 矩阵中逐阶段积累的 E2E 用例衔接。
- **E2E 工具**：**Playwright**（仓库根 `e2e/`）；首次运行前执行 **`pnpm test:e2e:install`**。

## 4. 技术约束与规范

### 1. 代码规范
*   严格遵循 **TypeScript 严格模式**，禁止使用 `any`。
*   前端组件必须**函数式**，逻辑复用抽取为 **Hooks**。
*   后端接口必须统一响应格式 `{ success, data, message }`。

### 2. 隐私合规
*   严禁在数据库明文存储密码（必须加盐哈希）。
*   严禁收集学生真实姓名、手机号、学校具体名称（除非必要且脱敏）。
*   注销功能必须物理删除或匿名化处理数据。

### 3. 性能要求
*   首屏 **FCP (First Contentful Paint) < 1.5s**。
*   **端到端实时延时（学生提交 -> 教师可见）&lt; 1s**（局域网/良好网络下）。

### 4. 兼容性
*   必须通过 **iOS Safari (最新 2 版)** 和 **Android Chrome (最新 2 版)** 测试。
*   必须适配 **375px** 宽度手机屏幕至 **1920px** 桌面屏幕。

## 5. 风险管理 (Risk Management)

| 风险点 | 可能性 | 影响程度 | 应对策略 |
| :--- | :---: | :---: | :--- |
| **高并发导致 WS 崩溃** | 中 | 高 | 采用 Redis Adapter 支撑多实例 WebSocket；限制单房间最大人数；实施降级策略 (轮询)。 |
| **学生恶意刷数据** | 中 | 中 | 后端增加频率限制 (Rate Limiting)；同一设备/IP 短时间多次提交需验证码。 |
| **移动端兼容性问题** | 高 | 中 | 早期介入真机测试；使用标准 Web API，避免实验性特性。 |
| **内容伦理争议** | 低 | 高 | 所有文案需经心理学专业人士审核；显著位置标注“娱乐参考，非医疗诊断”。 |
| **Admin/CMS 与自适应范围膨胀** | 中 | 中 | MVP 严格按 T2.7/T4.5–T4.8 验收；复杂编辑器等延后二期；优先「能发布、能回滚文案/题库」。 |
