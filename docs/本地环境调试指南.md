## 1. 你需要先准备什么？

| 工具                             | 作用（一句话）                                             |
| :------------------------------- | :--------------------------------------------------------- |
| **Node.js**（建议 LTS，如 20.x） | 运行前端、后端和 Prisma 脚本                               |
| **pnpm**                         | 本仓库规定的包管理器，用来安装依赖、跑脚本                 |
| **Docker Desktop**               | 在本机用「容器」跑 PostgreSQL、Redis，不用自己装数据库软件 |
| **Git**                          | 克隆仓库、拉代码                                           |

安装方式（任选其一即可）：

- **Node.js**：从 [https://nodejs.org](https://nodejs.org) 下载 LTS 安装包，一路下一步。
- **pnpm**：装好 Node 后，在终端执行：
  ```bash
  npm install -g pnpm
  ```
- **Docker Desktop（Windows）**：从 [https://www.docker.com/products/docker-desktop/](https://www.docker.com/products/docker-desktop/) 下载安装，安装后**重启电脑**（若安装程序提示）。首次打开 Docker Desktop 要等右下角图标变成「正在运行」。

**说明**：没有 Docker 也可以在本机单独安装 PostgreSQL，但本仓库已提供 `docker-compose.yml`，新手用 Docker 最省事，下文按 Docker 写法说明。

## 2. Docker 是干什么的？为什么要「起数据库」？

- **Docker**：把应用和它依赖的环境打成「镜像」，在容器里运行，避免「我电脑上能跑、你电脑上缺库」的问题。
- **docker compose**：用一个配置文件（本仓库根目录的 `docker-compose.yml`）**一次启动多个服务**（这里是 **PostgreSQL 数据库** 和 **Redis**）。
- **起数据库**：让 PostgreSQL 在后台跑起来，监听一个端口（默认 **5432**），你的 API（NestJS）和 Prisma 才能连上库、做迁移、跑集成测试。

本项目的 `docker-compose.yml` 里约定：

- 数据库服务名：**`db`**（镜像 `postgres:16`）
- 默认映射到本机端口：**5432**（若被占用可改，见下文）
- 默认账号（与 `.env.example` 一致）：
  - 用户：`ppa`
  - 密码：`ppa`
  - 库名：`ppa`

## 4. 用 Docker Compose 启动数据库（和 Redis）

### 4.1 确认 Docker 已运行

打开 **Docker Desktop**，等到状态为 **Running**（运行中）。若未运行，后面的 `docker` 命令会报错。

### 4.2 启动服务（后台运行）

在**项目根目录**执行：

```powershell
docker compose up -d
```

**这条命令在做什么？**

- `docker compose`：读取当前目录下的 `docker-compose.yml`。
- `up`：创建并启动里面定义的服务（`db`、`redis`）。
- `-d`：detached，在后台跑，不占用当前终端一直刷日志。

**只启动数据库（不启 Redis）**也可以（与仓库注释一致）：

```powershell
docker compose up -d db
```

Redis 目前主要用于后续 WebSocket / 限流等；若你只做数据库相关开发，可先只起 `db`。

### 4.3 查看是否在跑

```powershell
docker compose ps
```

**说明**：应能看到 `db`（以及若执行了完整 `up` 时的 `redis`）状态为 **running** 或 **healthy**。

查看 Postgres 健康检查日志（可选）：

```powershell
docker compose logs db
```

## 5. 连接字符串 `DATABASE_URL` 是什么？怎么写？

Prisma（`prisma/schema.prisma`）通过环境变量 **`DATABASE_URL`** 连接 PostgreSQL。

### 5.1 格式说明（看懂即可）

标准形式：

```text
postgresql://用户名:密码@主机:端口/数据库名?schema=public
```

与本项目 **Docker 默认配置**对应时，主机用 **`localhost`**（本机），端口 **`5432`**（若未改端口），即：

```text
postgresql://ppa:ppa@localhost:5432/ppa?schema=public
```

**各部分含义**：

| 片段             | 含义                                             |
| :--------------- | :----------------------------------------------- |
| `ppa`（第一个）  | 数据库用户名 `POSTGRES_USER`                     |
| `ppa`（密码）    | 数据库密码 `POSTGRES_PASSWORD`                   |
| `localhost`      | 本机；Docker 把容器端口映射到本机                |
| `5432`           | 本机映射端口，须与 `docker-compose` 实际映射一致 |
| `/ppa`           | 数据库名 `POSTGRES_DB`                           |
| `?schema=public` | 使用默认 `public` schema（Prisma 常用写法）      |

### 5.2 把连接串写进环境变量文件

仓库提供了 **`.env.example`**，你需要自己建 **`.env`**（不要提交到 Git，一般已在 `.gitignore` 里）。

1. 在**仓库根目录**复制一份：

   **PowerShell：**

   ```powershell
   Copy-Item .env.example .env
   ```

2. 用编辑器打开根目录的 **`.env`**，确认或增加一行（与上面默认 Docker 一致时）：

   ```env
   DATABASE_URL="postgresql://ppa:ppa@localhost:5432/ppa?schema=public"
   ```

3. Prisma 也会在 `prepare` 时读 **`prisma/.env`**（若存在）。为减少混乱，建议 **根目录 `.env` 与 `prisma/.env` 中的 `DATABASE_URL` 保持一致**，或只维护一处并在文档里说明。最小做法：**至少保证执行 `pnpm prisma migrate` 时能读到 `DATABASE_URL`**（Prisma CLI 会从 `prisma/schema.prisma` 所在目录向上找 `.env`；以你本机实际为准）。

**说明**：根目录 `.env` 里还可配置 `PORT`、`JWT_SECRET` 等，见 `.env.example` 注释。

## 7. 数据库结构迁移（Prisma Migrate）

第一次连上数据库后，需要把表结构建进去：

```powershell
pnpm exec prisma migrate dev
```

**这条命令在做什么？**

- 读取 `prisma/schema.prisma` 和 `prisma/migrations/` 里的迁移文件。
- 在数据库里执行 SQL，使表结构与团队一致。
- 开发时常会问你迁移名称；若只是同步已有迁移，按提示操作即可。

若你只想**应用已有迁移、不开发新迁移**（更接近生产部署），可用：

```powershell
pnpm exec prisma migrate deploy
```

**说明**：`migrate dev` 适合开发；`migrate deploy` 适合 CI/生产。

### 8.4 Playwright（学生端 E2E）

**前置条件**：

- 数据库已启动且迁移已应用（与 **8.3** 相同）。
- 首次在本机跑 Playwright 前建议安装浏览器（只需一次）：

```powershell

```

**PR / 本地门禁建议（测评相关改动）**：若改动触及以下路径，建议至少在本地跑一次 **`pnpm test:e2e`**（或在 CI 上跑同等命令）：

- `apps/web/src/app/test/**`
- `apps/web/src/hooks/use-*-test.ts`
- `apps/web/src/lib/progress-*.ts`

## 9. 端口被占用时怎么办？

若本机 **5432** 已被其他 PostgreSQL 占用，`docker-compose.yml` 支持用环境变量改映射端口。

1. 在**仓库根目录**的 **`.env`** 中增加（示例改为 **15432**）：

   ```env
   PPA_POSTGRES_PORT=15432
   ```

2. 重启容器：

   ```powershell
   docker compose down
   docker compose up -d
   ```

3. **`DATABASE_URL` 里的端口必须改成同一个**，例如：

   ```env
   DATABASE_URL="postgresql://ppa:ppa@localhost:15432/ppa?schema=public"
   ```

**说明**：`docker-compose.yml` 里写的是 `'${PPA_POSTGRES_PORT:-5432}:5432'`，左边是本机端口，右边是容器内端口（固定 5432）。你只改左边的映射即可。

Redis 同理，使用 **`PPA_REDIS_PORT`**（默认 **6379**），若改了端口，`REDIS_URL` 也要改成一致。

## 10. 常见问题

### 10.1 Prisma `P1001` / `Can't reach database server at 127.0.0.1:5432`

表示：**本机在 `DATABASE_URL` 所写的主机与端口上，连不上 PostgreSQL**（与 Prisma 无关，是网络/端口问题）。

按顺序排查：

1. **容器是否在跑且状态正常**  
   `docker compose ps` 里 `db` / `ppa-postgres` 应为 **running** 且 **healthy**。

2. **宿主机端口是否真正映射到容器 5432（常见根因）**  
   执行：

   ```powershell
   docker port ppa-postgres 5432
   ```

   - **期望**看到类似 `0.0.0.0:5432`（表示本机 5432 转发进容器）。
   - **若没有任何输出**：端口未发布到本机，可先重建：
     ```powershell
     docker compose down
     docker compose up -d
     docker port ppa-postgres 5432
     ```

3. **本机 5432 是否真的能连通**

   ```powershell
   Test-NetConnection -ComputerName 127.0.0.1 -Port 5432
   ```

   若 `TcpTestSucceeded` 为 **False**：检查是否有**其它程序占用 5432**（例如本机已安装的 PostgreSQL）：

   ```powershell
   netstat -ano | findstr :5432
   ```

   若 5432 被占用：不要与占用进程抢端口，改用 **`PPA_POSTGRES_PORT`** 换端口（见第 9 节），并同步修改 **`prisma/.env` 与根目录 `.env` 中的 `DATABASE_URL` 端口**，再 `docker compose down` / `up -d` 后重试 `pnpm exec prisma migrate dev`。

4. **`DATABASE_URL` 端口与 Docker 映射一致**  
   若使用默认映射，连接串应为 `...@127.0.0.1:5432/...` 或 `...@localhost:5432/...`（与第 5.2、第 9 节一致）。

5. **Docker Desktop 与防火墙**  
   确认 Docker Desktop 为运行中；若仍不通，可暂时排查防火墙/安全软件是否拦截对本机端口的访问。

### 10.2 `docker compose` 提示找不到命令

- 确认已安装 **Docker Desktop** 且已启动。
- 新版 Docker 推荐使用 **`docker compose`（带空格）**；若你用的是旧独立插件 `docker-compose`，功能类似，可把命令写成 `docker-compose up -d`（带横杠）。

### 10.3 想清空数据库重来（慎用）

会删掉容器内数据卷中的数据，**仅开发环境**考虑：

```powershell
docker compose down -v
docker compose up -d
```

然后重新执行 **`pnpm exec prisma migrate dev`**（或 `migrate deploy`）。

### 10.4 在 PowerShell 里敲了 `DATABASE_URL=...` 报错「无法识别」

**原因**：`VAR=value` 是 **Bash** 的写法；**Windows PowerShell** 不支持把环境变量写在命令前面这样赋值，整段会被当成命令名。

**推荐**：不要依赖终端临时变量，把连接串写进项目根目录 **`.env`** 里的 `DATABASE_URL=`（见第 5.2 节），Prisma / Node 会自动加载。

**若必须在当前 PowerShell 会话里临时设置**（仅本次窗口有效）：

```powershell
$env:DATABASE_URL = "postgresql://ppa:ppa@localhost:5432/ppa?schema=public"
```

查看是否生效：`echo $env:DATABASE_URL`

### 10.5 只粘贴了 `postgresql://ppa:...` 这一串也报错「无法识别」

**原因**：连接串**不是命令**，只是配置里要用的一段**文字**。单独粘贴到 PowerShell 时，系统会把它当成要运行的程序名，所以同样会 `CommandNotFoundException`。

**正确做法**：不要单独在终端里粘贴 URL。请把**整行**写进 **`.env`**（变量名 + 等号 + 引号包住的 URL），见第 5.2 节；或用上一节的 `$env:DATABASE_URL = "..."` 赋给环境变量后再执行 `pnpm` / `prisma`。

### 10.6 浏览器打开 `/test/standard` 报「Failed to fetch」、无法加载测评进度

**原因**：页面在 **`http://localhost:3000`**，接口默认请求 **`http://127.0.0.1:3001`**，二者**不同源**。浏览器会做跨域（CORS）检查；若 API 未对该来源返回允许头，前端 `fetch` 会失败，界面常显示 **`Failed to fetch`**（与「没起 API」的表现类似，但 Docker / `pnpm dev:api` 可能都是正常的）。

**处理**：

1. 确认 **API 已启动**（终端里有 `Nest application successfully started`）。
2. 使用**已启用 CORS** 的 API 版本：仓库在 `apps/api/src/main.ts` 为本地默认放行 `http://localhost:3000` 与 `http://127.0.0.1:3000`。修改后请 **重启 `pnpm dev:api`**，再刷新测评页。
3. 若 Web 跑在**其它端口**，在 API 使用的环境变量中设置 **`CORS_ORIGIN`**（逗号分隔多个源），例如根目录 `.env` 中：`CORS_ORIGIN=http://localhost:5173`，见仓库根 **`.env.example`** 注释。
